# Replay Attack Prevention - Complete Guide

## What is a Replay Attack?

A **replay attack** occurs when an attacker intercepts a valid authentication message and retransmits it later to gain unauthorized access.

### Simple Example

```
NORMAL AUTHENTICATION:
━━━━━━━━━━━━━━━━━━━━━━━
t=0s   Alice sends: POST /login with (username, nonce, signature)
       ✓ Server verifies signature
       ✓ Server creates session
       ✓ Alice logs in

REPLAY ATTACK:
━━━━━━━━━━━━━━━
Attacker (Eve) intercepts message and records it:
       (username="alice", nonce="abc123", signature="sig789")

t=5s   Eve resends: POST /login with SAME data
       ? Server sees: username="alice", nonce="abc123", signature="sig789"
       ? Signature is still valid (Alice signed it!)
       ? Nonce is still fresh (only 5 seconds old)
       ✗ Server re-authenticates Eve as Alice! [SECURITY BREACH]
```

## Three-Layer Replay Attack Protection

This system implements **3 independent layers** of protection:

### LAYER 1: Unique Nonce Per Request

**What:**
- Each authentication attempt gets a NEW unique 256-bit random nonce
- Nonce is generated by server and stored in database
- Client must sign this specific nonce

**How It Prevents Replay:**
```
t=0s   Login Attempt 1:
       Server generates: nonce_1 = random(256 bits)
       Client signs: sig_1 = sign(alice + nonce_1)
       ✓ Auth succeeds with (alice, nonce_1, sig_1)

t=10s  Replay Attack:
       Attacker sends: (alice, nonce_1, sig_1)
       Problem: Server doesn't want nonce_1 anymore!
       Server expects NEW nonce for next login
       ✓ But wait... also protected by other layers below

t=15s  New Login Attempt 2:
       Server generates: nonce_2 = random(256 bits)
       Attacker's old sig_1 invalid for nonce_2 (different message!)
       ✗ Attacker needs: sig_2 = sign(alice + nonce_2)
       ✗ But attacker doesn't have Alice's private key!
       ✗ BLOCKED
```

**Implementation:**
```python
# Each nonce is unique
nonce = QuantumSafeSignature.generate_nonce()  # 256-bit random
# Server stores: nonce value in database
nonce_obj = Nonce(user_id=user.id, nonce=nonce_value, ...)
```

### LAYER 2: Nonce Expiration (30 Second Window)

**What:**
- Each nonce valid for only 30 seconds
- After creation, nonce expires and becomes unusable
- Server checks: current_time < (created_time + 30s)

**Why 30 Seconds?**
```
Too short (<5s):    High false-reject rate (network delays)
Too long (>2min):   Large replay window, risk exposure

Goldilocks: 30 seconds
- Realistic for internet latency
- Tight enough to limit replay window
- Generous enough for mobile/slow networks
```

**How It Prevents Replay:**
```
SCENARIO: Late Replay Attack

t=0s   Normal Auth:
       Server creates: nonce_1, expires_at = 0 + 30s
       Client signs: sig_1 = sign(alice + nonce_1)
       ✓ Auth succeeds

t=2s   Immediate replay:
       Attacker sends: (alice, nonce_1, sig_1)
       Check: 2s < 30s? YES ✓ (still in window)
       Check: used? NO ✓ (not marked yet)
       Would authenticate... BUT WAIT!
       Layer 3 protection kicks in (see below)

t=35s  Late replay (after expiration):
       Attacker sends: (alice, nonce_1, sig_1)
       Check: 35s < 30s? NO ✗ (expired!)
       ✗ BLOCKED - Nonce expired!
       Message: \"Invalid or expired nonce\"

t=40s  New login:
       Server generates: nonce_2 = random(256 bits)
       nonce_1 completely gone from active rotation
       Attacker's old (nonce_1, sig_1) forever invalid
```

**Implementation:**
```python
# Configuration
NONCE_EXPIRY = timedelta(seconds=30)

# Server-side check
expires_at = datetime.utcnow() + NONCE_EXPIRY
nonce_obj = Nonce(..., expires_at=expires_at)

# Verification
if datetime.utcnow() > nonce_obj.expires_at:
    return "Nonce expired"
```

**Real-World Timeline:**
```
0.5s   ✓ User generates key pair
1.5s   ✓ User requests nonce
2.0s   ✓ Server generates nonce_1 (expires at 32s)
2.2s   ✓ User signs nonce_1
2.5s   ✓ User sends signature
3.0s   ✓ Server verifies, creates session
31.0s  ✗ Eve tries to replay: expired!
32.1s  ✗ Eve tries again: permanently invalid
```

### LAYER 3: Single-Use Enforcement (Used Flag)

**What:**
- Database stores boolean `used` flag (default: False)
- After first successful authentication with nonce, mark `used = True`
- Any second attempt with same nonce is rejected

**How It Prevents Replay:**
```
THE CLASSIC REPLAY ATTACK:

t=0s   Normal Auth:
       Server creates: nonce_1, used=False
       Client signs: sig_1 = sign(alice + nonce_1)
       ✓ Auth succeeds
       ✓ Server marks: nonce_1.used = True

t=2s   IMMEDIATE REPLAY (same request):
       Attacker sends: (alice, nonce_1, sig_1)
       Check: nonce_1 exists? YES ✓
       Check: nonce_1 expired? NO ✓ (2s < 30s)
       Check: nonce_1.used? YES ✗✗✗ CAUGHT!
       ✗ BLOCKED - \"Nonce already used\"
       
       Even though:
       - Signature is still valid ✓
       - Nonce is still in time window ✓
       - Message format is correct ✓
       
       The used flag catches it anyway!
```

**Implementation:**
```python
# Verification: Check if nonce is valid
def is_valid(self):
    return not self.is_expired() and not self.used

# After successful auth
nonce_obj.used = True
db.session.commit()

# On replay attempt
if not nonce_obj.is_valid():  # used=True causes this to fail
    return "Invalid or expired nonce"
```

## Combined Protection Example

### Attack Scenarios & Outcomes

#### Scenario 1: Immediate Replay
```
t=0s   User logs in with nonce_1
       Server: used=False, not expired ✓ AUTH
       
t=1s   Attacker replays with nonce_1
       Server: used=True ✓ NOT used=False ✗
       Result: BLOCKED ✗ (Layer 3)
```

#### Scenario 2: After Expiration
```
t=0s   User logs in with nonce_1 (expires at 30s)
       Server: used=False, not expired ✓ AUTH
       
t=35s  Attacker replays with nonce_1
       Server: used=False ✓ (first use), expired ✗
       Result: BLOCKED ✗ (Layer 2)
```

#### Scenario 3: Different Nonce
```
t=0s   User logs in with nonce_1
       Server: used=False, not expired ✓ AUTH
       
t=5s   Attacker replays with nonce_1
       Server: used=True ✗
       Result: BLOCKED ✗ (Layer 3)
       
       Even if attacker somehow got old nonce:
       - Each login generates NEW nonce
       - Old nonce_1 signatures won't verify for nonce_2
       - Would need private key to sign nonce_2
```

#### Scenario 4: Man-in-the-Middle Modifies Data
```
t=0s   Alice: (alice, nonce_1, sig_1) where sig_1 = sign(alice + nonce_1)
       Eve intercepts and modifies: (bob, nonce_1, sig_1)
       
       Server checks:
       - message = bob + nonce_1
       - sig_1 = sign(alice + nonce_1)
       - Do they match? NO ✗
       Result: BLOCKED ✗ (Signature verification)
       
       Even before checking replay protection!
```

```python
# SECURITY vs USABILITY trade-off

# Very tight (5 seconds)
NONCE_EXPIRY = timedelta(seconds=5)
# Pro: Minimizes replay window
# Con: May reject legitimate users with slow networks

# Recommended (30 seconds)
NONCE_EXPIRY = timedelta(seconds=30)
# Balance: Good security with reasonable usability

# Relaxed (2 minutes)
NONCE_EXPIRY = timedelta(seconds=120)
# Pro: Very user-friendly
# Con: Large replay window if key leaked
```

### Monitoring Replay Attempts

**Server logs show:**
```
[REPLAY] Failed nonce check for alice: exists=True, expired=False, used=True
```

**Metrics to track:**
- `used=True` rejections → Classic replay attempts
- `expired=True` rejections → Late replays
- `exists=False` rejections → Nonce forgery attempts

## Best Practices

### For Developers

1. **Always mark nonce as used** after auth succeeds
   ```python
   nonce_obj.used = True
   db.session.commit()
   ```

2. **Check nonce validity before signature verification**
   ```python
   if not nonce_obj.is_valid():
       return "Nonce invalid"
   # Only then verify signature
   ```

3. **Log replay attempts**
   ```python
   current_app.logger.warning(f"Replay attempt detected for {username}")
   ```

4. **Use tight expiration times** (30s is good default)

### For Users

1. **Don't reuse login credentials**
   - Each login requires fresh nonce
   - Each nonce is single-use

2. **Keep private key secure**
   - Attacker with private key can create valid signatures
   - But still can't replay (nonce is single-use!)

3. **Use HTTPS in production**
   - Prevents man-in-the-middle from intercepting nonce+signature

## Mathematical Proof of Protection

### Attack Probability Analysis

```
P(successful replay) = P(nonce exists) × P(not expired) × P(not used)

Normal flow:
- P(nonce exists) = 1.0 (server just created it)
- P(not expired) = 0.0 (server just marked used=True)
- P(not used) = 0.0 (server marks after auth)

P(successful replay) = 1.0 × 0.0 × 0.0 = 0 ✓

Even if all conditions met:
- Attacker needs private key to create valid signature
- P(has private key) ≈ 0 (different from public key!)
- P(successful forgery) ≈ 0

Result: P(successful replay) ≈ 0 ✓✓✓
```

## Frequently Asked Questions

### Q1: Why 30 seconds?
**A:** Balance between security and usability. Covers network latency (~5s) with safety margin (~25x). After 30s, nonce is permanently invalid.

### Q2: What if user loses connection during auth?
**A:** Nonce expires after 30s. User starts fresh login request for new nonce.

### Q3: Can attacker change the nonce?
**A:** Yes, but then the signature won't match (signature binds username+nonce). Attacker can't create new signature without private key.

### Q4: What if attacker has the private key?
**A:** They can create valid signatures, BUT:
- Each nonce single-use (used flag blocks replay)
- Each login needs fresh nonce (Layer 1)
- Nonce expires (Layer 2)
- All three must be true simultaneously - impossible!

### Q5: Is 30 seconds enough?
**A:** Yes. Replay attacks typically require:
- Network capture (< 1s)
- Analysis (< 5s)
- Replay attempt (< 30s)
- Successful before expiration (must be before 30s)

If attacker waits, nonce expires!

### Q6: What about old servers with clock skew?
**A:** Nonce uses server's current time. If server clock is wrong, nonce lifetime calculation is wrong, but same for all users so doesn't cause replay attacks.
